## システム設計書: DevGatewayDNS

---

### 1. プロジェクト概要

**目的**: 社内LANにおいて、WiFi接続のスマホ等を含む全クライアントから、ホスト名ベースで仮想Webページにアクセス可能にする統合ツール。

**コンセプト**: リバースプロキシをメイン機能とし、プロキシ設定に連動してDNSレコードを自動生成する。加えて独立したDNSレコード管理機能も持つ。全機能をWeb UIで統合管理し、単一バイナリで配布する。DNS変更が困難なクライアント向けにフォワードプロキシも常時提供する。

### 5. 機能詳細

#### 5.1 リバースプロキシ（メイン機能）

**プロキシルール設定項目**:

| 項目 | 説明 | 例 |
|---|---|---|
| ホスト名 | クライアントがアクセスするFQDN | foo.local |
| バックエンドプロトコル | バックエンドへの転送プロトコル | http or https |
| 転送先IP | 自動（自端末）または手動（任意IP指定） | 自動 / 127.0.0.1 |
| 転送先ポート | バックエンドのポート | 8080 |
| 有効/無効 | ルールの有効化フラグ | true |

**転送先IPの設定**:

| 設定 | 動作 | ユースケース |
|---|---|---|
| 自動 | DevGatewayDNS動作端末のIPを使用 | バックエンドが同一端末上にある場合 |
| 手動 | ユーザーが指定した任意のIPへ転送 | バックエンドが別端末、またはlocalhost固定で動作している場合等 |

**対応パターン**:
- HTTP(:80)で受信 → バックエンドへHTTPまたはHTTPSで転送
- HTTPS(:443)で受信 → SSL終端後、バックエンドへHTTPまたはHTTPSで転送

**動作フロー**:
1. クライアントがDNSクエリを送信 → DevGatewayDNSがLISTENアドレスのIPを応答
2. クライアントが443/80にアクセス → SNI/Hostヘッダでルール照合
3. 該当ルールの転送先（IP:ポート）へリクエスト転送
4. バックエンドのレスポンスをクライアントに返却

**ヘッダ・Cookie処理**:

以下の処理はプロキシの透過性を保つために全ルールで自動的に適用される。個別の設定は不要。フォワードプロキシ経由のアクセスにも同様に適用される。

| 機能 | 説明 |
|---|---|
| ホストヘッダ転送 | クライアントの要求ホスト名をバックエンドにそのまま転送 |
| レスポンスヘッダ書き換え | バックエンドのLocationヘッダ等のURLを書き換え |
| Cookieドメイン置換 | Set-CookieのDomainをフロント側ホスト名に置換 |
| Cookieパス置換 | Set-Cookieのpathを必要に応じて書き換え |
| X-Forwarded-* 付与 | X-Forwarded-For / Host / Proto を自動付与 |

**LISTENポート設定**:
- HTTP受付ポート: デフォルト80（設定変更可）
- HTTPS受付ポート: デフォルト443（設定変更可）

#### 5.2 DNSサーバー

**自動レコード（プロキシ連動、メモリ管理）**:

プロキシルールに連動するAレコードはDBには保存せず、メモリ上で管理する。起動時にproxy_rules（enabled=true）とNIC情報からメモリ上にmapを構築する。

**メモリ上のデータ構造**: `map[ホスト名] → []IPアドレス`

**起動時の構築処理**:
1. proxy_rules（enabled=true）を全件読み込み
2. LISTENアドレス設定とNIC情報からIPリストを決定
3. 各ルールに対してホスト名→IPリストのmapエントリを生成

**自動AレコードのIP決定ロジック**:

| 条件 | 動作 |
|---|---|
| LISTENが特定IP指定 | LISTEN IPのみ |
| LISTENが0.0.0.0 | 全NICのIP（NICごとに1件） |

**メモリ更新タイミング**:

| 操作 | メモリ処理 |
|---|---|
| アプリ起動 | 全ルールから構築 |
| ルール追加 | エントリ追加 |
| ルール削除 | エントリ削除 |
| ルール有効化 | エントリ追加 |
| ルール無効化 | エントリ削除 |
| ルール更新（ホスト名変更） | 旧ホスト名削除→新ホスト名追加 |

NIC IPはアプリ起動時に取得した値で固定。DHCP更新等でIPが変わった場合はアプリ再起動で再構築する。

**手動レコード管理（DB管理）**:

対応レコードタイプ: A / AAAA / CNAME / MX / TXT / SRV / NS / PTR / CAA / SOA / NAPTR / SSHFP / TLSA / DS / DNSKEY

Web UI上で自由に追加・編集・削除可能。

**DNS応答フロー**:
1. メモリmap検索（自動レコード）→ ヒットすればIPリストを応答して終了
2. DB検索（手動レコード）→ ヒットすれば値を応答して終了
3. クエリを受信したNICに対応する上流DNSへフォワード

同一ホスト名が自動・手動の両方に存在する場合は、手動レコードを優先する。

**上流DNSフォワーディング（NIC別）**:
- 起動時に各NICに設定されているDNSサーバーを取得し、NIC IP → 上流DNSの対応をメモリに保持
- DNSクエリを受信したNICを特定し、そのNICのDNSサーバーへ転送
- NICのDNS情報が取得できない場合はフォールバック上流DNS（デフォルト: 8.8.8.8 / 1.1.1.1）を使用
- Web UIからフォールバック上流DNSを設定可能

**DNSクエリ履歴（メモリ、リングバッファ）**:

DNSクエリログはDBには保存せず、メモリ上のリングバッファで直近の履歴を保持する。

| 項目 | 内容 |
|---|---|
| 保持件数 | デフォルト1000件（settingsで変更可能） |
| 保持データ | クエリ元IP、問い合わせホスト名、レコードタイプ、応答結果（自動/手動/上流転送）、応答時間 |
| 管理画面 | WebSocketでリアルタイム配信、ホスト名・クライアントIPでフィルタリング可能 |

**LISTENポート**: UDP/TCP 53（設定変更可）

#### 5.3 SSL証明書管理

**証明書の自動管理**:
- プロキシルール追加時にホスト名に対する自己署名証明書を自動生成
- ルート認証局（CA）証明書を初回起動時に生成
- 証明書の有効期限管理と自動更新
- Web UIからの証明書一覧表示、手動再生成

**CA証明書の配布**:

| 機能 | 内容 |
|---|---|
| ダウンロード | Web UIからPEM / DER / P12形式で取得 |
| 直接インストールURL | `http://{host}:9090/ca` にアクセスで証明書ファイルを返却。iOS/Androidはそのままインストールダイアログが開く |
| QRコード表示 | 上記URLのQRコードをWeb UIに表示。PC画面からスマホで読み取り |
| インストール手順ガイド | iOS / Android / Windows / macOS 別の手順をWeb UI内にテキストで表示 |

**HTTPS利用時の注意**: フロント側がHTTPSのルールを利用する場合、アクセス経路（DNS方式/フォワードプロキシ方式）に関わらずCA証明書のインストールが必要。HTTPのみであればCA証明書は不要。

#### 5.4 フォワードプロキシ

DNS設定変更が困難なクライアント（iOS等）向けに、HTTPプロキシとして設定するだけでDevGatewayDNSの全機能を利用可能にする。リバースプロキシ・DNSと常時同時稼働し、同一のプロキシルール・DNSレコードが適用される。ヘッダ・Cookie処理もリバースプロキシと同様に自動適用される。

**動作**:
1. クライアントがWiFiプロキシに `DevGatewayDNS_IP:8888` を設定
2. HTTPリクエスト → ホスト名をプロキシルールと照合、該当すれば転送先（IP:ポート）へ転送、該当しなければ外部へそのまま転送
3. HTTPS(CONNECT)リクエスト → プロキシルールに該当すればSSL終端して転送先へ転送（CA証明書インストール要）、該当しなければトンネルとして外部へ通過

**LISTENポート**: デフォルト8888（設定変更可）

#### 5.5 Web UI

**画面構成**:

| 画面 | 機能 |
|---|---|
| ダッシュボード | 稼働状況サマリ、NIC一覧、ポートステータス（バインド・ループバックチェック結果）、直近のアクセスログ |
| プロキシ設定 | ルール一覧、追加/編集/削除、有効/無効切替 |
| DNS管理 | 自動レコード一覧（ロック表示）、手動レコードCRUD、DNSクエリ履歴（リアルタイム） |
| 証明書管理 | 証明書一覧、CA証明書ダウンロード、QRコード表示、OS別インストール手順ガイド、手動再生成 |
| ステータスモニタ | リアルタイムHTTPアクセスログ（WebSocket）、トラフィック統計 |
| システム設定 | LISTENポート、フォワードプロキシポート、フォールバック上流DNS、管理UIポート、DNSクエリ履歴件数、ログレベル等 |

**ダッシュボードのヘルスチェック**: ポートごとにバインド成功/失敗およびループバック接続テスト結果を表示。画面表示時に自動チェック実行。手動リチェックボタンを提供。

**DNS管理画面のレコード表示**: メモリ上の自動レコード（locked=true, source="proxy"）とDB上の手動レコード（locked=false, source="manual"）をマージして一覧表示する。自動レコードは編集・削除不可。

**多言語対応**:
- 初期対応言語: 日本語(ja)、英語(en)
- ブラウザのAccept-Language優先順位に従い自動選択
- en-US, en-GB等のサブタグはenにフォールバック
- Web UIからの手動切替も可能
- 翻訳リソースはフロントエンドビルドに含まれバイナリに同梱
- 他言語は翻訳リソース追加で拡張可能な構造

**テーマ**: OSのカラースキーム設定（`prefers-color-scheme`）に自動追従し、ライト/ダークを切替。

**管理UIポート**: デフォルト9090（設定変更可）

#### 5.6 ネットワーク

- 起動時にホストの全NICを検出し、全インターフェースでListen
- 各NICに設定されているDNSサーバーも取得し上流DNS転送に使用
- NICの追加/削除、DHCP更新によるIP変更時は再起動で対応

### 6. CLI設計

CLIの出力は英語固定（多言語化しない）。

```
devgatewaydns serve [options]       # フォアグラウンドで起動
devgatewaydns install [options]     # OSサービスとして登録
devgatewaydns uninstall             # サービス登録解除
devgatewaydns start                 # 登録済みサービス開始
devgatewaydns stop                  # 登録済みサービス停止
devgatewaydns status                # サービス状態表示
devgatewaydns version               # バージョン表示
```

**起動時ヘルスチェック**: `serve` / `start` 実行時に全ポートのバインド成功確認 + ループバック接続テストを実施。失敗時はCLIに警告出力（起動は継続）。

**serve / install 共通オプション**:

| オプション | デフォルト | 説明 | 例 |
|---|---|---|---|
| `--http-port` | 80 | HTTP受付ポート | `--http-port 8080` |
| `--https-port` | 443 | HTTPS受付ポート | `--https-port 8443` |
| `--dns-port` | 53 | DNS受付ポート | `--dns-port 5353` |
| `--proxy-port` | 8888 | フォワードプロキシポート | `--proxy-port 3128` |
| `--admin-port` | 9090 | 管理UI用ポート | `--admin-port 3000` |
| `--listen` | 0.0.0.0 | LISTENアドレス（複数指定可） | `--listen 192.168.1.10 --listen 10.0.0.1` |
| `--db` | (バイナリ同ディレクトリ)/devgatewaydns.db | DBファイルパス | `--db /var/data/app.db` |

**設定の優先順位**:
- オプションなし → SQLiteの保存済み設定で起動
- オプションあり → 指定値で起動し、SQLiteにも上書き保存
- `--db` のみCLI専用（DB接続前に必要なため）

**使用例**:

```
# 保存済み設定で起動（初回はデフォルト値）
devgatewaydns serve

# ポート指定で起動（設定も上書き保存）
devgatewaydns serve --http-port 8080 --https-port 8443

# 特定NICのみでサービス登録
devgatewaydns install --listen 192.168.1.10 --listen 10.0.0.1

# DBパス指定
devgatewaydns serve --db /opt/devgatewaydns/data.db
```

`install` 時に指定したオプションはサービス登録時の起動引数として保存され、`start` で起動した際にその設定が適用される。

サービス管理は `kardianos/service` で全OS共通APIにより実装。

### 7. データモデル

#### 7.1 SQLite (WALモード)

- 読み取りが書き込みをブロックしないため、アクセスログの頻繁な書き込みとWeb UIからの読み取りが並行動作可能
- DB初期化時に `PRAGMA journal_mode=WAL` を実行
- DBファイルに加え `-wal` と `-shm` ファイルがローカルに生成される

**DBパス**:
- デフォルト: 実行バイナリと同ディレクトリの `devgatewaydns.db`
- CLIオプション `--db` で変更可能
- 初回起動時にDBファイルとテーブルを自動生成、デフォルト設定をinsert

**マイグレーション**:
- pressly/goose v3 により起動時に自動実行
- マイグレーションSQLファイルは `embed.FS` でバイナリに同梱
- 未適用のマイグレーションを順次適用、失敗時はロールバック

**テーブル一覧**:

| テーブル | 用途 |
|---|---|
| proxy_rules | プロキシルール |
| dns_records | 手動DNSレコード |
| ca_certificate | CA証明書（1行） |
| host_certificates | ホスト別SSL証明書 |
| settings | システム設定（key-value） |
| access_logs | HTTPアクセスログ |

**proxy_rules**:

| カラム | 型 | 説明 |
|---|---|---|
| id | INTEGER PK | 自動採番 |
| hostname | TEXT UNIQUE | ホスト名 (例: foo.local) |
| backend_protocol | TEXT | バックエンドへの転送プロトコル ("http" or "https") |
| backend_ip | TEXT | 転送先IP（NULL=自動、値あり=手動指定） |
| backend_port | INTEGER | 転送先ポート (例: 8080) |
| enabled | BOOLEAN | 有効/無効 |
| created_at | DATETIME | 作成日時 |
| updated_at | DATETIME | 更新日時 |

**dns_records（手動レコード専用）**:

| カラム | 型 | 説明 |
|---|---|---|
| id | INTEGER PK | 自動採番 |
| name | TEXT | FQDN (例: mail.example.local) |
| type | TEXT | レコードタイプ (A, AAAA, CNAME, MX等) |
| value | TEXT | レコード値 |
| ttl | INTEGER | TTL秒数 (デフォルト: 300) |
| priority | INTEGER | 優先度 (MX, SRV用、NULL可) |
| weight | INTEGER | 重み (SRV用、NULL可) |
| port | INTEGER | ポート (SRV用、NULL可) |
| created_at | DATETIME | 作成日時 |
| updated_at | DATETIME | 更新日時 |

自動レコード（プロキシ連動Aレコード）はDBには保存せず、メモリ上で管理する。

**ca_certificate（1行のみ）**:

| カラム | 型 | 説明 |
|---|---|---|
| id | INTEGER PK | 常に1 |
| cert_pem | BLOB | CA証明書PEM |
| key_pem | BLOB | CA秘密鍵PEM |
| expires_at | DATETIME | 有効期限 |
| created_at | DATETIME | 作成日時 |

**host_certificates**:

| カラム | 型 | 説明 |
|---|---|---|
| id | INTEGER PK | 自動採番 |
| hostname | TEXT UNIQUE | 対象ホスト名 |
| cert_pem | BLOB | 証明書PEM |
| key_pem | BLOB | 秘密鍵PEM |
| expires_at | DATETIME | 有効期限 |
| created_at | DATETIME | 作成日時 |

**settings**:

| カラム | 型 | 説明 |
|---|---|---|
| key | TEXT PK | 設定キー |
| value | TEXT | 設定値 (JSON) |
| updated_at | DATETIME | 更新日時 |

主な設定キー:

| key | 内容 | デフォルト |
|---|---|---|
| http_port | HTTP受付ポート | 80 |
| https_port | HTTPS受付ポート | 443 |
| dns_port | DNS受付ポート | 53 |
| proxy_port | フォワードプロキシポート | 8888 |
| admin_port | 管理UIポート | 9090 |
| listen_addresses | LISTENアドレスリスト | ["0.0.0.0"] |
| upstream_dns_fallback | NICのDNS取得不能時のフォールバック上流DNS | ["8.8.8.8","1.1.1.1"] |
| dns_query_history_size | DNSクエリ履歴保持件数 | 1000 |
| log_level | ログレベル | "info" |

**access_logs**:

| カラム | 型 | 説明 |
|---|---|---|
| id | INTEGER PK | 自動採番 |
| timestamp | DATETIME | アクセス日時 |
| source | TEXT | アクセス経路 ("reverse" or "forward") |
| client_ip | TEXT | クライアントIP |
| hostname | TEXT | リクエストホスト名 |
| method | TEXT | HTTPメソッド |
| path | TEXT | リクエストパス |
| status_code | INTEGER | レスポンスステータス |
| response_time_ms | INTEGER | 応答時間(ms) |
| backend | TEXT | 転送先 (ip:port) |

#### 7.2 メモリ管理データ

**自動DNSレコード**:

| 項目 | 内容 |
|---|---|
| データ構造 | `map[ホスト名] → []IPアドレス`（sync.RWMutexで保護） |
| 構築タイミング | アプリ起動時にproxy_rules + NIC情報から生成 |
| 更新 | ルール追加/削除/有効化/無効化/ホスト名変更に連動 |
| IP変更対応 | アプリ再起動で再構築 |

**NIC別上流DNS対応**:

| 項目 | 内容 |
|---|---|
| データ構造 | `map[NIC IP] → []上流DNSアドレス` |
| 構築タイミング | アプリ起動時に各NICのDNS設定を取得 |
| フォールバック | NICのDNS取得不能時はsettingsの `upstream_dns_fallback` を使用 |

**DNSクエリ履歴**:

| 項目 | 内容 |
|---|---|
| データ構造 | リングバッファ |
| 保持件数 | デフォルト1000件（settingsで変更可能） |
| 保持データ | クエリ元IP、問い合わせホスト名、レコードタイプ、応答結果（自動/手動/上流転送）、応答時間 |
| 配信 | WebSocketでリアルタイム配信 |

### 8. API設計 (REST + WebSocket)

**Base**: `http://localhost:9090/api/v1`

**プロキシ管理**:
- `GET /proxy/rules` — ルール一覧取得
- `POST /proxy/rules` — ルール追加（メモリ上の自動Aレコードも追加）
- `PUT /proxy/rules/:id` — ルール更新（メモリ上の自動Aレコードも更新）
- `DELETE /proxy/rules/:id` — ルール削除（メモリ上の自動Aレコードも削除）
- `PATCH /proxy/rules/:id/toggle` — 有効/無効切替（メモリ上の自動Aレコードも追加/削除）

**DNS管理**:
- `GET /dns/records` — レコード一覧（メモリ上の自動レコード + DB上の手動レコードをマージ、自動レコードにはlocked=trueフラグ付与）
- `POST /dns/records` — 手動レコード追加
- `PUT /dns/records/:id` — 手動レコード更新
- `DELETE /dns/records/:id` — 手動レコード削除
- `GET /dns/upstream` — フォールバック上流DNS設定取得
- `PUT /dns/upstream` — フォールバック上流DNS設定更新

**証明書管理**:
- `GET /certs` — 証明書一覧
- `POST /certs/:hostname/regenerate` — 手動再生成
- `GET /certs/ca/download` — CA証明書ダウンロード（PEM/DER/P12、クエリパラメータで形式指定）
- `GET /certs/ca/qrcode` — CA証明書インストールURLのQRコード画像取得

**CA証明書直接配布**（管理API外、ルート直下）:
- `GET /ca` — CA証明書ファイルを直接返却（iOS/Androidから直接アクセスしてインストール用）

**ステータス**:
- `GET /status/overview` — 稼働状況サマリ
- `GET /status/interfaces` — NIC一覧と状態（各NICの上流DNS情報含む）
- `GET /status/health` — ポート別ヘルスチェック実行（バインド・ループバック結果）
- `WebSocket /status/live` — リアルタイムHTTPアクセスログ配信
- `WebSocket /dns/queries/live` — リアルタイムDNSクエリ履歴配信

**システム設定**:
- `GET /settings` — 全設定取得
- `PUT /settings` — 設定更新（ポート変更等は再起動要求）

### 12. クライアント側の導入

**DNS方式（PC等）**:
1. WiFiルーターのDHCP設定でDNSサーバーをDevGatewayDNSのIPに設定（全端末に自動配布）、または各端末で手動DNS設定
2. HTTPS利用時はCA証明書をインストール

**フォワードプロキシ方式（iOS等）**:
1. WiFi設定 → プロキシ → 手動 → サーバー: DevGatewayDNSのIP / ポート: 8888
2. HTTPS利用時はCA証明書をインストール

**CA証明書インストール**:
1. Web UIの証明書管理画面でQRコードを表示
2. スマホでQRコードを読み取り → ブラウザで証明書URLにアクセス → インストール
3. またはブラウザから直接 `http://{host}:9090/ca` にアクセス
4. Web UI内にOS別（iOS / Android / Windows / macOS）のインストール手順ガイドを掲載
