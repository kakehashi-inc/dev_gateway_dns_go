## システム仕様: DevGatewayDNS

---

### 1. 概要

DevGatewayDNSは社内LANにおいて、WiFi接続のスマホ等を含む全クライアントから、ホスト名ベースで仮想Webページにアクセス可能にする統合ツールである。リバースプロキシをメイン機能とし、プロキシ設定に連動してDNSレコードを自動生成する。加えて独立したDNSレコード管理機能も持つ。全機能をWeb UIで統合管理し、単一バイナリで配布する。DNS設定変更が困難なクライアント向けにフォワードプロキシも常時提供する。

### 4. リバースプロキシ（メイン機能）

#### プロキシルール設定項目

| 項目 | 説明 | 例 |
|---|---|---|
| ホスト名 | クライアントがアクセスするFQDN | foo.local |
| バックエンドプロトコル | バックエンドへの転送プロトコル | http or https |
| 転送先IP | 自動（自端末）または手動（任意IP指定） | 自動 / 127.0.0.1 |
| 転送先ポート | バックエンドのポート | 8080 |
| 有効/無効 | ルールの有効化フラグ | true |

#### 転送先IPの設定

| 設定 | 動作 | ユースケース |
|---|---|---|
| 自動 (NULL) | DevGatewayDNS動作端末のIPを使用 | バックエンドが同一端末上にある場合 |
| 手動 | ユーザーが指定した任意のIPへ転送 | バックエンドが別端末にある場合等 |

#### 対応パターン

- HTTP(:80)で受信 → バックエンドへHTTPまたはHTTPSで転送
- HTTPS(:443)で受信 → SSL終端後、バックエンドへHTTPまたはHTTPSで転送

#### 動作フロー

1. クライアントがDNSクエリを送信 → DevGatewayDNSがLISTENアドレスのIPを応答
2. クライアントが443/80にアクセス → SNI/Hostヘッダでルール照合
3. 該当ルールの転送先（IP:ポート）へリクエスト転送
4. バックエンドのレスポンスをクライアントに返却

#### ヘッダ・Cookie自動処理

全ルールで自動適用。フォワードプロキシ経由のアクセスにも同様に適用。

| 機能 | 説明 |
|---|---|
| ホストヘッダ転送 | クライアントの要求ホスト名をバックエンドにそのまま転送 |
| レスポンスヘッダ書き換え | バックエンドのLocationヘッダ等のURLを書き換え |
| Cookieドメイン置換 | Set-CookieのDomainをフロント側ホスト名に置換 |
| Cookieパス置換 | Set-Cookieのpathを必要に応じて書き換え |
| X-Forwarded-* 付与 | X-Forwarded-For / Host / Proto を自動付与 |

#### LISTENポート設定

- HTTP受付ポート: デフォルト80（設定変更可）
- HTTPS受付ポート: デフォルト443（設定変更可）

### 5. DNSサーバー

#### 自動レコード（プロキシ連動、メモリ管理）

プロキシルールに連動するAレコードはDBには保存せず、メモリ上で管理する。

**メモリ上のデータ構造**: `map[ホスト名] → []IPアドレス`（sync.RWMutexで保護）

**起動時の構築処理**:

1. proxy_rules（enabled=true）を全件読み込み
2. LISTENアドレス設定とNIC情報からIPリストを決定
3. 各ルールに対してホスト名→IPリストのmapエントリを生成

**自動AレコードのIP決定ロジック**:

| 条件 | 動作 |
|---|---|
| LISTENが特定IP指定 | LISTEN IPのみ |
| LISTENが0.0.0.0 | 全NICのIP（NICごとに1件） |

**メモリ更新タイミング**:

| 操作 | メモリ処理 |
|---|---|
| アプリ起動 | 全ルールから構築 |
| ルール追加 | エントリ追加 |
| ルール削除 | エントリ削除 |
| ルール有効化 | エントリ追加 |
| ルール無効化 | エントリ削除 |
| ルール更新（ホスト名変更） | 旧ホスト名削除→新ホスト名追加 |

NIC IPはアプリ起動時に取得した値で固定。DHCP更新等でIPが変わった場合はアプリ再起動で再構築する。

#### 手動レコード管理（DB管理）

対応レコードタイプ: A / AAAA / CNAME / MX / TXT / SRV / NS / PTR / CAA / SOA / NAPTR / SSHFP / TLSA / DS / DNSKEY

Web UI上で自由に追加・編集・削除可能。

#### DNS応答フロー

1. DB検索（手動レコード）→ ヒットすれば値を応答して終了
2. メモリmap検索（自動レコード）→ ヒットすればIPリストを応答して終了
3. クエリを受信したNICに対応する上流DNSへフォワード

同一ホスト名が自動・手動の両方に存在する場合は、手動レコードを優先する。

#### 上流DNSフォワーディング（NIC別）

- 起動時に各NICに設定されているDNSサーバーを取得し、NIC IP → 上流DNSの対応をメモリに保持
- DNSクエリを受信したNICを特定し、そのNICのDNSサーバーへ転送
- NICのDNS情報が取得できない場合はフォールバック上流DNS（デフォルト: 8.8.8.8 / 1.1.1.1）を使用
- Web UIからフォールバック上流DNSを設定可能

**NIC別上流DNSデータ構造**: `map[NIC IP] → []上流DNSアドレス`

#### DNSクエリ履歴（メモリ、リングバッファ）

DNSクエリログはDBには保存せず、メモリ上のリングバッファで直近の履歴を保持する。

| 項目 | 内容 |
|---|---|
| 保持件数 | デフォルト1000件（settingsで変更可能） |
| 保持データ | クエリ元IP、問い合わせホスト名、レコードタイプ、応答結果（自動/手動/上流転送）、応答時間 |
| 管理画面 | WebSocketでリアルタイム配信、ホスト名・クライアントIPでフィルタリング可能 |

**LISTENポート**: UDP/TCP 53（設定変更可）

### 6. SSL証明書管理

#### 証明書の自動管理

- ルートCA証明書を初回起動時に自動生成（有効期限10年、ECDSA P-256）
- プロキシルール追加時にホスト名に対する自己署名証明書を自動生成（有効期限1年）
- 証明書の有効期限管理と自動更新
- Web UIからの証明書一覧表示、手動再生成

#### CA証明書の配布

| 機能 | 内容 |
|---|---|
| ダウンロード | Web UIからPEM / DER / P12形式で取得 |
| 直接インストールURL | `http://{host}:9090/ca` にアクセスで証明書ファイルを返却。iOS/Androidはそのままインストールダイアログが開く |
| QRコード表示 | 上記URLのQRコードをWeb UIに表示。PC画面からスマホで読み取り |
| インストール手順ガイド | iOS / Android / Windows / macOS 別の手順をWeb UI内にテキストで表示 |

### 7. フォワードプロキシ

DNS設定変更が困難なクライアント（iOS等）向けに、HTTPプロキシとして設定するだけでDevGatewayDNSの全機能を利用可能にする。リバースプロキシ・DNSと常時同時稼働し、同一のプロキシルール・DNSレコードが適用される。ヘッダ・Cookie処理もリバースプロキシと同様に自動適用される。

**動作**:

1. クライアントがWiFiプロキシに `DevGatewayDNS_IP:8888` を設定
2. HTTPリクエスト → ホスト名をプロキシルールと照合、該当すれば転送先（IP:ポート）へ転送、該当しなければ外部へそのまま転送
3. HTTPS(CONNECT)リクエスト → プロキシルールに該当すればSSL終端して転送先へ転送（CA証明書インストール要）、該当しなければトンネルとして外部へ通過

**LISTENポート**: デフォルト8888（設定変更可）

### 10. CLI設計

CLIの出力は英語固定（多言語化しない）。

#### コマンド一覧

| コマンド | 説明 |
|---|---|
| `devgatewaydns serve [options]` | フォアグラウンドで起動 |
| `devgatewaydns install [options]` | OSサービスとして登録 |
| `devgatewaydns uninstall` | サービス登録解除 |
| `devgatewaydns start` | 登録済みサービス開始 |
| `devgatewaydns stop` | 登録済みサービス停止 |
| `devgatewaydns status` | サービス状態表示 |
| `devgatewaydns version` | バージョン表示 |

#### 起動オプション（serve / install 共通）

| オプション | デフォルト | 説明 |
|---|---|---|
| `--http-port` | 80 | HTTP受付ポート |
| `--https-port` | 443 | HTTPS受付ポート |
| `--dns-port` | 53 | DNS受付ポート |
| `--proxy-port` | 8888 | フォワードプロキシポート |
| `--admin-port` | 9090 | 管理UIポート |
| `--listen` | 0.0.0.0 | LISTENアドレス（複数指定可） |
| `--db` | (バイナリ同ディレクトリ)/devgatewaydns.db | DBファイルパス |

#### 設定の優先順位

- オプションなし → SQLiteの保存済み設定で起動
- オプションあり → 指定値で起動し、SQLiteにも上書き保存
- `--db` のみCLI専用（DB接続前に必要なため）

`install` 時に指定したオプションはサービス登録時の起動引数として保存され、`start` で起動した際にその設定が適用される。サービス管理は `kardianos/service` で全OS共通APIにより実装。

#### 起動時ヘルスチェック

`serve` / `start` 実行時に全ポートのバインド成功確認 + ループバック接続テストを実施。失敗時はCLIに警告出力（起動は継続）。

### 12. データベース

#### SQLite (WALモード)

- 読み取りが書き込みをブロックしないため、アクセスログの頻繁な書き込みとWeb UIからの読み取りが並行動作可能
- DB初期化時に `PRAGMA journal_mode=WAL` を実行
- DBファイルに加え `-wal` と `-shm` ファイルがローカルに生成される

**DBパス**:
- デフォルト: 実行バイナリと同ディレクトリの `devgatewaydns.db`
- CLIオプション `--db` で変更可能
- 初回起動時にDBファイルとテーブルを自動生成、デフォルト設定をinsert

**マイグレーション**:
- pressly/goose v3 により起動時に自動実行
- マイグレーションSQLファイルは `embed.FS` でバイナリに同梱
- 未適用のマイグレーションを順次適用、失敗時はロールバック

**テーブル一覧**: テーブルの詳細定義は `Documents/テーブル定義.md` を参照。

| テーブル | 用途 |
|---|---|
| proxy_rules | プロキシルール |
| dns_records | 手動DNSレコード |
| ca_certificate | CA証明書（1行） |
| host_certificates | ホスト別SSL証明書 |
| settings | システム設定（key-value） |
| access_logs | HTTPアクセスログ |

#### 日時の取り扱い

- DBへの記録: 全テーブルのタイムスタンプはUTCで記録する
- Web UIでの表示: ブラウザのロケールに従いローカル日時で表示する（`toLocaleString()` / `toLocaleDateString()`）
- アクセスログ: 日付+時刻（`toLocaleString()`）
- 証明書の有効期限・作成日: 日付のみ（`toLocaleDateString()`）

#### アクセスログ管理

アクセスログはプロキシルールに定義されているホストへのリクエストのみ記録する（ルールに該当しないリクエストは記録しない）。

**自動削除**:

- アプリ起動時および日付変更時（毎日0時）に、設定された保持期間（デフォルト7日）を超えた古いログを自動削除する
- 削除基準: 現在日付 - (保持日数 - 1) より前のタイムスタンプを持つレコードを削除
- 起動時はログ削除後にVACUUMを実行し、DBファイルサイズを最適化してからサーバーを起動する（VACUUMはDB全体のメンテナンスとして起動時のみ実行）
- 保持日数はWeb UIのシステム設定画面から変更可能（settings: `access_log_retention_days`）

#### メモリ管理データ

| データ | 構造 | 構築タイミング |
|---|---|---|
| 自動DNSレコード | `map[ホスト名] → []IPアドレス`（sync.RWMutex） | 起動時にproxy_rules + NIC情報から生成、ルール変更時に連動更新 |
| NIC別上流DNS | `map[NIC IP] → []上流DNSアドレス` | 起動時に各NICのDNS設定を取得 |
| DNSクエリ履歴 | リングバッファ（デフォルト1000件） | WebSocketでリアルタイム配信 |

### 14. ビルド

**パッケージマネージャ**: Yarn（npmは使用しない）

**Makefile ビルドターゲット**:

| ターゲット | 内容 |

| `frontend` | Yarnによるフロントエンドビルド → `frontend/dist/` |
| `backend` | Goビルド → 6プラットフォーム向け単一バイナリ |

### 15. クライアント側の導入

#### DNS方式（PC等）

1. WiFiルーターのDHCP設定でDNSサーバーをDevGatewayDNSのIPに設定（全端末に自動配布）、または各端末で手動DNS設定
2. HTTPS利用時はCA証明書をインストール

#### フォワードプロキシ方式（iOS等）

1. WiFi設定 → プロキシ → 手動 → サーバー: DevGatewayDNSのIP / ポート: 8888
2. HTTPS利用時はCA証明書をインストール

#### CA証明書インストール

1. Web UIの証明書管理画面でQRコードを表示
2. スマホでQRコードを読み取り → ブラウザで証明書URLにアクセス → インストール
3. またはブラウザから直接 `http://{host}:9090/ca` にアクセス
4. Web UI内にOS別（iOS / Android / Windows / macOS）のインストール手順ガイドを掲載

**HTTPS利用時の注意**: フロント側がHTTPSのルールを利用する場合、アクセス経路（DNS方式/フォワードプロキシ方式）に関わらずCA証明書のインストールが必要。HTTPのみであればCA証明書は不要。
