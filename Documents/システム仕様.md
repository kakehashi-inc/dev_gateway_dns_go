## システム仕様: DevGatewayDNS

---

### 1. 概要

DevGatewayDNSは社内LANにおいて、WiFi接続のスマホ等を含む全クライアントから、ホスト名ベースで仮想Webページにアクセス可能にする統合ツールである。リバースプロキシをメイン機能とし、プロキシ設定に連動してDNSレコードを自動生成する。DNS設定変更が困難なクライアント向けにフォワードプロキシも常時提供する。

---

### 2. 起動と停止

#### CLI コマンド

| コマンド | 説明 |
|---|---|
| `devgatewaydns serve [options]` | フォアグラウンドで起動 |
| `devgatewaydns install [options]` | OSサービスとして登録 |
| `devgatewaydns uninstall` | サービス登録解除 |
| `devgatewaydns start` | 登録済みサービス開始 |
| `devgatewaydns stop` | 登録済みサービス停止 |
| `devgatewaydns status` | サービス状態表示 |
| `devgatewaydns version` | バージョン表示 |

#### 起動オプション (serve / install 共通)

| オプション | デフォルト | 説明 |
|---|---|---|
| `--http-port` | 80 | HTTP受付ポート |
| `--https-port` | 443 | HTTPS受付ポート |
| `--dns-port` | 53 | DNS受付ポート |
| `--proxy-port` | 8888 | フォワードプロキシポート |
| `--admin-port` | 9090 | 管理UIポート |
| `--listen` | 0.0.0.0 | LISTENアドレス (複数指定可) |
| `--db` | (バイナリ同ディレクトリ)/devgatewaydns.db | DBファイルパス |

#### 設定の優先順位

1. CLIオプション指定あり: 指定値で起動、SQLiteにも上書き保存
2. CLIオプション指定なし: SQLiteの保存済み設定で起動
3. 初回起動時: デフォルト値を使用しSQLiteに保存

---

### 3. リバースプロキシ

#### 動作フロー

1. クライアントがDNSクエリ送信 → DevGatewayDNSがLISTENアドレスのIPを応答
2. クライアントが443/80にアクセス → SNI/Hostヘッダでルール照合
3. 該当ルールの転送先 (IP:ポート) へリクエスト転送
4. バックエンドのレスポンスをクライアントに返却

#### ヘッダ・Cookie自動処理

全ルールで自動適用。フォワードプロキシ経由のアクセスにも同様に適用。

- ホストヘッダ転送: クライアントの要求ホスト名をバックエンドにそのまま転送
- レスポンスヘッダ書き換え: Locationヘッダ等のURLをフロント側に書き換え
- Cookieドメイン置換: Set-CookieのDomainをフロント側ホスト名に置換
- X-Forwarded-* 付与: X-Forwarded-For / Host / Proto を自動付与

---

### 4. DNSサーバー

#### 自動レコード (メモリ管理)

プロキシルール (enabled=true) に連動するAレコードはメモリ上の `map[ホスト名]→[]IP` で管理する。

- 構築: アプリ起動時にproxy_rules + NIC情報から生成
- 更新: ルール追加/削除/有効化/無効化/ホスト名変更に連動

#### 手動レコード (DB管理)

dns_recordsテーブルで管理。Web UIから追加・編集・削除可能。

対応レコードタイプ: A / AAAA / CNAME / MX / TXT / SRV / NS / PTR / CAA / SOA / NAPTR / SSHFP / TLSA / DS / DNSKEY

#### DNS応答優先順位

1. 手動レコード (DB) → ヒットすれば応答
2. 自動レコード (メモリ) → ヒットすれば応答
3. 上流DNSフォワード → NIC別に対応する上流DNSへ転送

#### 上流DNSフォワーディング

- 各NICのDNS設定を起動時に取得しメモリに保持
- DNS取得不能時はフォールバック上流DNS (デフォルト: 8.8.8.8, 1.1.1.1) を使用

#### DNSクエリ履歴

メモリ上のリングバッファで直近の履歴を保持 (デフォルト1000件)。WebSocketでリアルタイム配信。

---

### 5. SSL証明書管理

- ルートCA証明書を初回起動時に自動生成 (有効期限10年)
- プロキシルール追加時にホスト別証明書を自動生成 (有効期限1年)
- Web UIからCA証明書のダウンロード (PEM/DER形式)
- QRコード表示によるモバイル端末への配布
- `/ca` エンドポイントによる直接配布

---

### 6. フォワードプロキシ

DNS設定変更が困難なクライアント (iOS等) 向け。HTTPプロキシとして設定するだけで全機能を利用可能。

- HTTP: ホスト名をプロキシルールと照合、該当すれば転送、非該当は外部へ転送
- HTTPS (CONNECT): 該当すればSSL終端して転送、非該当はトンネルとして通過

---

### 7. REST API

Base: `http://localhost:9090/api/v1`

#### プロキシ管理

| メソッド | パス | 説明 |
|---|---|---|
| GET | /proxy/rules | ルール一覧取得 |
| POST | /proxy/rules | ルール追加 |
| PUT | /proxy/rules/:id | ルール更新 |
| DELETE | /proxy/rules/:id | ルール削除 |
| PATCH | /proxy/rules/:id/toggle | 有効/無効切替 |

#### DNS管理

| メソッド | パス | 説明 |
|---|---|---|
| GET | /dns/records | レコード一覧 (自動+手動マージ) |
| POST | /dns/records | 手動レコード追加 |
| PUT | /dns/records/:id | 手動レコード更新 |
| DELETE | /dns/records/:id | 手動レコード削除 |
| GET | /dns/upstream | フォールバック上流DNS取得 |
| PUT | /dns/upstream | フォールバック上流DNS更新 |

#### 証明書管理

| メソッド | パス | 説明 |
|---|---|---|
| GET | /certs | 証明書一覧 |
| POST | /certs/:hostname/regenerate | 手動再生成 |
| GET | /certs/ca/download | CA証明書ダウンロード |
| GET | /certs/ca/qrcode | CA証明書QRコード画像 |

#### ステータス

| メソッド | パス | 説明 |
|---|---|---|
| GET | /status/overview | 稼働状況サマリ |
| GET | /status/interfaces | NIC一覧と状態 |
| GET | /status/health | ヘルスチェック実行 |
| WebSocket | /status/live | リアルタイムアクセスログ |
| WebSocket | /dns/queries/live | リアルタイムDNSクエリ履歴 |

#### システム設定

| メソッド | パス | 説明 |
|---|---|---|
| GET | /settings | 全設定取得 |
| PUT | /settings | 設定更新 (ポート変更は再起動要) |

#### CA証明書直接配布

| メソッド | パス | 説明 |
|---|---|---|
| GET | /ca | CA証明書ファイルを直接返却 |

---

### 8. データベース

- SQLite WALモード (読み取りが書き込みをブロックしない)
- 初回起動時にDBファイルとテーブルを自動生成
- goose v3によるマイグレーション (起動時自動実行)
- マイグレーションSQLはバイナリに同梱 (embed.FS)

---

### 9. ヘルスチェック

起動時に全ポートのバインド確認 + ループバック接続テストを実施。失敗時は警告出力 (起動は継続)。
管理API `/api/v1/status/health` でも実行可能。

---

### 10. 起動要件

- 管理者権限で実行 (ポート53/80/443のLISTENに必要)
- Windows: 管理者として実行
- Linux/macOS: root または `setcap`
